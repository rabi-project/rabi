# Tangle — Test & Verification Plan (MVP through Production)
### v1.0 · July 2026 · Agent-feedable. Companion to `mvp-build-plan.md` and `full-build-plan.md`.

---

## Instructions to the coding agent (read first)

1. **Tests ship with the feature, never after.** A milestone PR without its suites green is
   incomplete by definition.
2. **Never weaken a criterion to pass it.** If a bar in this document seems wrong, stop and flag
   it as a plan question; do not edit this file to make CI green.
3. **A flaky test is a failed test.** Deflake (root-cause: seed it, fake the clock, remove the
   sleep) or delete it with a recorded decision — never retry-until-green.
4. **Determinism is a tested property, not a hope.** Anything seeded runs twice in CI; outputs
   must be byte-identical.
5. **Goldens are code.** Changing a golden placement file requires the PR label
   `golden-change` and a one-line justification per changed scenario.
6. Wall-clock budget: the default CI pipeline stays **under 15 minutes** (unit + component +
   one e2e smoke). Slow suites (bench, soak, chaos, live-vendor) run on schedules or labels —
   never block the inner loop.

## 0. The evidence ladder — what testing is *for* on this project

Every suite in this plan exists to earn one rung. This ladder is also the honest answer to
"what does the MVP prove":

| Rung | Claim | Earned by | When |
|---|---|---|---|
| **E1 — It runs** | A vendor-neutral fleet control plane exists and operates end-to-end | The compose demo, executed by a stranger from the quickstart | MVP |
| **E2 — It is correct** | Behavior matches the spec: lifecycle, protocol, validation, accounting invariants | Unit + property + conformance + integration suites | MVP, then every phase |
| **E3 — It is better** | Calibration-aware fleet placement beats current practice on fidelity, SLO compliance, wait | The benchmark: real calibration data, seeded, open, CI-verified statistics | MVP |
| **E4 — It survives production** | Correctness and value hold under failure, load, upgrade, attack, and time | Chaos, load, soak, migration, security suites + pilot/production SLOs + probes | Phases 1–2 |
| **E5 — It is independently adopted** | Others implement/deploy it and pass conformance | The public conformance program; a second implementation certifying | Phases 2–3 |

The MVP claims **E1+E2+E3 and nothing more** — the boundary is stated in the benchmark report,
which is what makes the claims it *does* make irrefutable (see `bench/report.md` methodology
requirements, §2.M6).

## 1. The test pyramid (applies to every phase)

| Layer | Scope | Tooling | Runs |
|---|---|---|---|
| L0 Static | lint, vet, proto breaking-change check, schema meta-validation, SPDX/license audit | golangci-lint, ruff, buf breaking, jsonschema CLI | every PR |
| L1 Unit | pure logic: state machine, filter/score, ESP math, quota/ledger arithmetic, validation | go test, pytest; property tests via rapid (Go) / hypothesis (Py) | every PR |
| L2 Component | API vs real Postgres (testcontainers); registry vs fake adapters; adapter conformance vs Aer; vendor adapters vs recorded cassettes | testcontainers, conformance harness, VCR-style record/replay | every PR |
| L3 End-to-end | compose stack job round-trips; operator on kind; quickstart-docs test | docker compose, kind, scripted README execution | every PR (smoke) + nightly (full) |
| L4 System qualities | determinism, load, soak, chaos, upgrade/migration, security (fuzz, authz matrix, CVE gate) | k6/vegeta, custom chaos runner, go-fuzz/atheris, grype | nightly / weekly / release |
| L5 Production verification | canary probe jobs, estimator-calibration tracking, shadow scheduling, SLOs + error budgets, continuous conformance, benchmark-as-regression | built into the product + release pipeline | continuously at pilot/prod sites |

Two Tangle-specific ideas run through everything:

**The physics oracle.** A scheduler whose value claim is "better fidelity" needs ground truth.
On simulators, ground truth is exact (ideal vs. noisy distributions → total-variation/Hellinger
fidelity — this is why the replay fleet is the testing backbone). On real hardware, ground truth
for interesting circuits is unknowable — so production verification uses **probe jobs with
analytically known outputs** (Bell/GHZ and mirror-style circuits): cheap, continuous, per-target
canaries that measure both the *device* and the *scheduler's predictions* (§4).

**Conformance is the contract test.** The same public suite that certifies third-party drivers
(spec `conformance/README.md`, 8 categories) runs in CI against our own adapters first. We are
never our own exception.

## 4. Phase 2 (production) — the E4 suites

**Chaos suite** (weekly + before every release; pass = invariants hold, not "nothing broke"):
kill adapter mid-RUNNING · partition control-plane↔agent 5 min · Postgres failover mid-bind ·
duplicate LISTEN/NOTIFY delivery · replay-clock skew · adapter returns garbage protobuf ·
vendor cassette 10× latency · disk-full on ledger write. **Invariants asserted after every
scenario:** no job lost (every accepted job reaches a queryable state), no duplicate execution
(idempotency ledger), terminal states immutable, usage ≤ caps + tolerance, audit trail gapless.

| Area | Success criteria |
|---|---|
| HA failover | RTO < 60 s, RPO = 0 (committed writes survive); drill automated, runs weekly |
| Zero-downtime upgrade | N-1→N with live synthetic workload: zero jobs failed *attributable to upgrade*; API unavailability < 30 s; rollback rehearsed |
| Migration goldens | DB snapshot from every released version migrates forward; invariants suite green on each |
| Load | 10,000 pending jobs, 100 targets: scheduler cycle p99 < 2 s; API read p99 < 300 ms, write p99 < 1 s; no unbounded queue growth |
| Soak | 72 h with accelerated replay: RSS growth < 5%/24 h post-warmup; goroutines bounded; zero non-terminal jobs older than policy max |
| Security | fuzz program-format parsers ≥ 1M execs, zero crashes/hangs; CVE gate: no HIGH/CRITICAL at release; secrets never in logs (log-scan test); external pen test before first production sign-off |
| Mutation testing (scheduler + fsm packages, quarterly) | mutation score ≥ 65% — tests must kill logic mutants, not just touch lines |
| **Production SLOs** | availability ≥ 99.5%; error budgets tracked; probe coverage: every target probed ≥ every 15 min; estimator drift alert if 7-day median calibration error worsens by > 0.05 |

**Production verification machinery (built into the product, this phase):**
- **Canary probes:** known-output circuits (Bell/GHZ/mirror-style) auto-submitted per target;
  results feed device health, SLO dashboards, and estimator-calibration tracking. Probes are
  first-class tenant `system/probes` with budget caps — visible in accounting like everyone else.
- **Shadow scheduling:** candidate policies run in shadow (compute placements, don't execute);
  counterfactual quality estimated via probe-calibrated ESP; a policy is promotable only after
  ≥ 2 weeks of shadow superiority — this is how scheduling evolves in production without risk.
- **Benchmark-as-regression:** every release re-runs the public benchmark; fidelity/SLO/wait
  regressions > 5% vs previous release block the release unless RFC-annotated.
- **Continuous conformance:** nightly against every in-tree adapter and any vendor endpoint we
  hold credentials for; public dashboard — the program eats its own dog food before certifying others.

## 6. What the MVP proves, and why it is irrefutable (normative for the report)

The MVP makes exactly three claims, each with its own proof mechanism:

- **P1 (existence, E1):** a vendor-neutral control plane scheduling declarative, multi-modal
  jobs across heterogeneous targets through one adapter protocol *exists and runs*. Proof:
  constructive — `docker compose up` on the reviewer's own machine. Existence proofs cannot be
  argued with, only executed.
- **P2 (correctness, E2):** its behavior matches a published spec. Proof: the suites in §2 +
  the same public conformance harness that will certify third parties, green in public CI.
- **P3 (value, E3):** calibration-aware fleet placement beats documented current practice
  (static best-device selection, per the Ravi et al. behavioral evidence) on fidelity and SLO
  compliance at realistic load. Proof: the benchmark — real vendor calibration baselines,
  disclosed synthetic drift, exact simulator ground truth, ≥5 seeds with CIs, byte-identical
  reruns, open data + code.

Irrefutability is a property of the *method*, not the size of the numbers: every claim is
reproducible offline by a stranger, every known objection is pre-empted in the methodology
(drift is synthetic — disclosed and justified; fidelity is simulator-measured — disclosed, with
the real-hardware pathway feature-flagged; baselines are the real status quo — cited), and the
benchmark carries no built-in success criterion, so it is falsifiable in the only way that
matters: someone can run it and lose the argument with data, or decline to run it and concede
the method. The MVP explicitly does **not** claim E4 (production survival) or E5 (adoption) —
stating that boundary in the report is itself part of what makes P1–P3 unassailable.
